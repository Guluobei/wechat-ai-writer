## 1. 项目背景与目标

### 1.1 定位

构建一个 skill，帮助用户实现公众号文章创作的全流程自动化（含关键节点人工确认）。聚焦 AI 行业，覆盖产品、技术、科普、行业解读四类内容，未来可扩展至 AI+ 方向。

### 1.2 目标用户

AI 行业自媒体创作者，需要高频产出高质量、有时效性、有差异化的公众号内容。

### 1.3 核心成功标准

- 覆盖从选题到发布后复盘的完整闭环（10 个模块 + 1 个配置层）
- AI 行业特有能力：差异化分析、风格辨识度、数据图表、反馈闭环
- 关键节点可控：大纲确认、初稿确认、发布前确认三个门控点
- 一体化产出：可直接粘贴公众号编辑器的 HTML + 发布元信息
- 模块可独立调用，也可串联成全流程

## 2. 整体架构

### 2.1 架构选型

采用**方案二：主 Skill 编排 + 独立子模块**。

理由：
1. AI 行业内容迭代快，模块需可独立升级（信源、配图工具、排版规则会变）
2. 关键节点确认天然需要独立编排层管理状态流转
3. 独立模块支持单独调用（如只排版、只调研），投入产出比最高

### 2.3 关键确认节点

| 节点 | 位置 | 确认内容 | 未通过的处理 |

| 确认点 1 | 大纲构建后 | 文章结构、论点覆盖、配图标注 | 修改大纲或回退调研 |
| 确认点 2 | 初稿撰写后 | 内容质量、调性、事实准确性 | 修改初稿或回退大纲 |
| 确认点 3 | 发布前 | 成品 HTML、封面、标题、摘要、标签 | 修改任一元素或回退排版 |

## 4. 各模块详细设计

### 模块 0：主编排器（Orchestrator）

**职责**：全流程指挥中心，不碰内容生产，只管流程。

**核心能力**：
- **状态机管理**：维护文章从"选题"到"已发布"到"已复盘"的完整生命周期
- **关键节点门控**：三个确认点暂停，等待用户反馈后推进
- **中间产物缓存**：每个环节产出存入 history，支持回退到任意环节重做
- **数据传递**：统一 ArticleProject 对象在模块间流转
- **单模块调用**：支持跳过流程，直接调用指定子模块（如"只排版"）
- **配置加载**：启动时从配置层加载风格档案、信源清单、排版模板

**接口**：
- 输入：用户指令（选题关键词 / 文章项目 ID + 目标环节）
- 输出：流程状态更新 + 当前环节产出

---

### 模块 1：选题策划（Topic Planner）

**职责**：从 AI 行业热点中找到值得写的选题，打磨成可执行的选题方案。

**核心能力**：
- **热点追踪**：扫描固定信源（AI 媒体、X/Twitter KOL、论文站、产品发布）+ Web 搜索，识别近期热点
- **选题评估**：对候选选题打分——时效性、话题热度、受众价值、差异化空间，筛出 Top N
- **角度建议**：不只给话题，给切入角度（如某新模型发布 → 技术解读 / 产品分析 / 行业影响）
- **选题方案输出**：标题（暂定）、核心论点、目标读者、预计篇幅、重点调研方向

**接口**：
- 输入：领域关键词 / 空输入（自动扫描热点）
- 输出：选题方案文档（写入 `topic` 字段）

---

### 模块 2：调研收集（Researcher）

**职责**：围绕选题收集素材，整理成带引用的结构化调研报告。

**核心能力**：
- **双通道采集**：
  - 固定信源通道：扫描预置 AI 行业信源清单，按选题关键词匹配
  - 临时搜索通道：Web 搜索补充热点和背景
- **信源权威性分级**：一手官方 / 权威媒体 / 社区讨论 / 个人观点
- **竞品内容扫描**：搜索同主题已发布的公众号/自媒体文章，识别已覆盖角度和内容空白点
- **信息去重与冲突检测**：多源报道去重，事实矛盾标记待核实
- **结构化整理**：按"事实 / 观点 / 数据 / 案例"四类归类，每条附引用出处（来源名 + 链接 + 检索时间）
- **调研报告输出**：选题背景、关键事实、核心观点、可用数据、典型案例、竞品覆盖分析、引用列表

**接口**：
- 输入：选题方案（`topic` 字段）
- 输出：调研报告（写入 `research` 字段，含 `competitor_coverage` 和 `citations`）

---

### 模块 3：大纲构建（Outliner）★ 确认点 1

**职责**：基于调研素材设计文章结构，产出可指导撰写的大纲。

**核心能力**：
- **结构生成**：根据文章类型匹配结构模板，生成多级大纲
- **逻辑校验**：论证链完整性检查——论点是否有素材支撑、关键维度是否遗漏、过渡是否自然
- **节奏设计**：标注每节预计篇幅占比、阅读重点、配图类型（ai_generated / screenshot / data_chart / none）
- **钩子设计**：开头钩子（抓注意力）+ 结尾收束（引导互动/关注）
- **差异化标注**：基于竞品扫描结果，在大纲中标注"本文独有的角度/信息"，确保与已有内容有差异

**接口**：
- 输入：调研报告（`research` 字段）
- 输出：多级大纲（写入 `outline` 字段）
- 门控：输出后暂停，等待用户确认

---

### 模块 4：正文撰写（Writer）★ 确认点 2

**职责**：按大纲和素材生成符合 AI 行业调性的初稿。

**核心能力**：
- **调性适配**：根据文章类型切换风格——技术解读偏严谨精确、产品分析偏洞察锐利、科普偏深入浅出、行业解读偏格局视野
- **风格档案参照**：始终参照配置层的风格档案（用词偏好、句式习惯、观点倾向、禁忌表述），保证跨文章调性一致
- **素材织入**：事实/数据/案例自然织入正文，不堆砌不失真，引用以公众号友好方式呈现
- **专业术语处理**：AI 术语密集，自动判断哪些需解释、哪些默认读者懂，关键术语首次出现加注
- **实时事实核查**：写作过程中实时核对调研报告引用，确保正文陈述有出处支撑，不让错误事实进入初稿
- **过渡与节奏**：控制段落长度和信息密度，注意可读性和叙事节奏

**接口**：
- 输入：大纲（`outline`）+ 调研报告（`research`）+ 风格档案（配置层）
- 输出：完整初稿 Markdown（写入 `draft` 字段，含 `inline_citations`）
- 门控：输出后暂停，等待用户确认

---

### 模块 5：配图制作（Image Designer）

**职责**：生成封面图和文中插图，三通道混合（AI 生成 / 截图 / 数据图表）。

**核心能力**：
- **配图规划**：根据大纲配图标注，逐张判断用 AI 生成 / 截图 / 数据图表，输出配图清单
- **AI 生成通道**：
  - 封面图：根据主题和风格偏好生成，强调视觉冲击力
  - 概念图/示意图：抽象概念（模型架构、技术流程）生成示意图
  - 氛围图：段落间视觉缓冲，科技感插画
- **截图通道**：
  - 产品展示类：识别文章提到的 AI 产品/工具，生成截图指引（截什么、重点突出什么）
  - 技术演示类：代码/界面演示，生成截图规范
- **数据图表通道**：
  - 基准测试对比图：模型性能对比
  - 能力雷达图：多维度能力展示
  - 市场数据图：行业数据可视化
  - 自动从调研报告的 data_points 提取数据生成
- **图文匹配校验**：检查图片与对应段落内容是否契合
- **图片管理**：统一命名、存储路径、尺寸规格（公众号特定尺寸）

**接口**：
- 输入：大纲配图标注（`outline.sections[].image_*`）+ 初稿（`draft`）+ 调研数据（`research.data_points`）
- 输出：配图清单 + 图片文件（写入 `images` 字段）

---

### 模块 6：排版美化（Formatter）

**职责**：初稿 + 配图 → 公众号编辑器可直接粘贴的 HTML。

**核心能力**：
- **Markdown → 公众号 HTML**：转换正文，全部内联样式（公众号不支持外部 CSS）
- **内置排版模板**：预置 AI 行业风格模板（标题、引用块、代码块、图片说明、分隔线），支持切换
- **图片嵌入**：配图按规划插入正确位置，处理图文混排和图片说明
- **引用呈现**：内联引用标记 → 公众号友好样式（脚注式 / 文末参考式，用户可选）
- **移动端适配**：字号、行距、段落间距、图片尺寸按移动端优化
- **预览自检**：格式错乱检查、图片缺失检查、引用完整性检查

**接口**：
- 输入：初稿（`draft`）+ 配图（`images`）+ 排版模板（配置层）
- 输出：成品 HTML（写入 `formatted` 字段）

---

### 模块 7：校对优化（Reviewer）

**职责**：成品 HTML 的最后一道质量把关（文字 + 可读性 + 合规）。事实核查已在撰写环节前置完成。

**核心能力**：
- **文字校对**：错别字、标点、语法、术语一致性
- **可读性优化**：信息密度检查、长段拆分建议、过渡生硬检测，给出具体修改建议
- **合规检查**：AI 行业敏感点——避免模型能力夸大表述、避免未经验证预测、敏感话题（AI 安全、监管）措辞谨慎
- **问题分级**：必须修 / 建议修 / 可选优化，支持快速决策

**接口**：
- 输入：成品 HTML（`formatted`）
- 输出：校对报告（写入 `review` 字段）
- 异常流：如问题涉及正文改动，回退到撰写环节重做受影响部分

---

### 模块 8：发布分发（Publisher）★ 确认点 3

**职责**：生成发布所需的全部元信息，辅助完成发布。

**核心能力**：
- **封面元信息**：从配图模块封面图生成缩略图版本，确认最终封面
- **标签生成**：根据内容生成公众号标签（3-5 个），覆盖 AI 细分领域关键词
- **摘要撰写**：生成文章摘要（120 字内），准确概括且吸引点击
- **发布时机建议**：基于 AI 行业读者活跃时段规律 + 复盘模块历史数据，给出推荐发布时间
- **标题优化**：3-5 个候选标题（直述型 / 悬念型 / 数字型），供用户选
- **发布前清单**：成品 HTML、封面图、摘要、标签、标题、发布时间，逐项确认

**接口**：
- 输入：成品 HTML（`formatted`）+ 配图（`images`）+ 复盘历史经验（配置层）
- 输出：发布就绪包（写入 `publish` 字段）
- 门控：输出后暂停，等待用户确认

---

### 模块 9：复盘反馈（Retrospective）

**职责**：发布后收集数据，分析表现，沉淀经验回喂选题和发布模块。

**核心能力**：
- **数据收集**：发布后 N 天（默认 7 天）收集阅读量、完读率、转发数、留言数
- **表现分析**：按内容类型、选题来源、标题风格、发布时间段维度分析哪些组合表现最佳
- **经验沉淀**：
  - 回喂选题模块：哪类选题/角度表现好 → 调整选题评分权重
  - 回喂发布模块：哪个时间段/标题风格点击率高 → 优化时机和标题建议
  - 回喂风格档案：读者反馈中反映的调性偏好 → 更新风格档案
- **趋势追踪**：长期追踪数据趋势，识别读者偏好变化

**接口**：
- 输入：文章项目 ID + 发布后数据（用户手动提供或对接公众号数据 API）
- 输出：复盘报告（写入 `retrospective` 字段）+ 更新配置层历史经验

---

### 配置层（Config Layer）

**职责**：跨文章持久化的配置和经验，所有模块共享。

**核心原则：开箱即用 + 渐进增强。** 每项配置都有内置默认值，首次使用即可跑通全流程；用户后续可按需替换为自己的配置，但不强制。

**组成**：

| 配置项 | 内置默认值（开箱即用） | 用户自定义（可选） | 更新时机 |
|--------|----------------------|-------------------|----------|
| 风格档案 | AI 行业通用调性：专业但不晦涩、客观陈述、避免夸大 | 用户提供 3-5 篇满意历史文章，skill 提取个人风格覆盖默认值 | 首次使用后渐进配置 + 复盘模块自动建议 |
| 信源清单 | 内置 20+ AI 行业权威信源（Hugging Face Blog、Anthropic/OpenAI 官方博客、arXiv、TechCrunch AI 等） | 用户增删信源、调整优先级 | 随时 |
| 排版模板 | 内置 1 套 AI 行业通用模板（深色标题 + 代码块高亮 + 引用块样式） | 用户自定义模板或调整配色/字号 | 随时 |
| 历史经验 | 空（无历史数据时，发布时机建议用通用时段规律） | 复盘模块自动写入 | 每篇文章复盘后自动更新 |

**冷启动保护**：任何模块启动时，若配置缺失，自动加载内置默认值并在产出中标注"使用默认配置，建议后续个性化"。绝不因配置缺失而中断流程。

## 5. 典型工作流

### 5.1 全流程（从选题到复盘）

```
用户：「写一篇关于 GPT-5 发布的解读」
  ↓
[模块1] 扫描热点 → 输出选题方案
  ↓
[模块2] 固定信源 + Web搜索 + 竞品扫描 → 输出调研报告（含引用）
  ↓
[模块3] 生成大纲 → ★ 用户确认大纲
  ↓ （确认通过）
[模块4] 参照风格档案 + 实时事实核查 → 输出初稿 → ★ 用户确认初稿
  ↓ （确认通过）
[模块5] AI生成封面 + 数据图表 + 截图指引 → 输出配图
  ↓
[模块6] 初稿+配图 → 公众号HTML（内联样式，移动端适配）
  ↓
[模块7] 文字+可读性+合规校对 → 输出校对报告
  ↓ （如有必修项 → 回退撰写；否则继续）
[模块8] 封面缩略图+标签+摘要+标题候选+时机建议 → ★ 用户确认发布包
  ↓ （确认通过）
用户复制HTML到公众号编辑器，上传封面，填入元信息，发布
  ↓
[模块9] 7天后收集数据 → 分析表现 → 沉淀经验回喂配置层
```

## 7. 健壮性与错误处理设计

> 本章解决"流程跑通 + 出错看得懂"的核心需求。面向非技术用户，所有错误必须自解释。

### 7.1 统一错误呈现规范

每个错误信息必须包含三部分，缺一不可：

```
❌ 【发生了什么】调研环节：Web 搜索暂时无法连接
   【为什么】可能是网络波动或搜索服务暂时不可用
   【你该做什么】已自动切换到固定信源通道继续调研。
                 如果固定信源也失败，请稍后重试，
                 或直接告诉我「跳过调研，用我提供的素材」。
```

**原则**：
- 永远用大白话，不用技术术语（不写"API timeout"、"null pointer"）
- 永远给一个可操作的下一步，绝不只报错不指路
- 能自动恢复的，恢复后告知用户"已自动处理"；不能自动恢复的，明确告诉用户怎么办

### 7.2 各模块降级方案

| 模块 | 可能失败点 | 自动降级 | 降级后用户感知 |
|------|-----------|---------|---------------|
| 选题策划 | Web 搜索失败 | 用固定信源扫描热点 | "网络搜索暂不可用，已用内置信源选了 3 个热点" |
| 调研收集 | Web 搜索失败 | 只用固定信源 | "Web 搜索暂时不可用，已用固定信源完成调研，部分热点可能未覆盖" |
| 调研收集 | 某信源抓取失败 | 跳过该信源，继续其他 | "XX 信源暂时无法访问，已跳过，其余信源正常" |
| 正文撰写 | 事实核查无引用支撑 | 标记该处为"待核实"并继续 | "第 3 段的数据未找到出处，已标注为待核实，发布前请确认" |
| 配图制作 | AI 图像生成失败 | 降级为纯文字占位 + 截图指引 | "封面图生成失败，已留占位符，你可以手动上传或稍后重试" |
| 配图制作 | 用户未提供截图 | 跳过该图，标注位置，不阻塞流程 | "产品截图需要你提供，已在文中标注位置，排版时不影响其他部分" |
| 配图制作 | 数据图表生成失败 | 降级为数据表格文本呈现 | "图表生成失败，已用文字表格替代，内容不丢失" |
| 排版美化 | 配图缺失 | 对应位置留空或文字占位 | "第 2 张配图缺失，已留占位，发布前可补图" |
| 校对优化 | 无严重问题 | 直接通过 | "校对完成，无必须修改项" |
| 发布分发 | 无历史数据（首次） | 用通用时段建议 | "暂无历史数据，建议工作日早 8 点或晚 8 点发布（行业通用规律）" |
| 复盘反馈 | 用户未提供数据 | 跳过复盘，提示下次提供 | "暂无数据，跳过复盘。发布 7 天后告诉我阅读数据，我帮你分析" |

### 7.3 回退与重做边界

当用户在确认点拒绝或校对环节发现问题时，回退规则如下：

| 触发场景 | 回退到哪 | 保留什么 | 重做什么 |
|---------|---------|---------|---------|
| 大纲确认不通过 | 模块 3（大纲） | 调研报告完整保留 | 只重做大纲 |
| 初稿确认不通过 | 模块 4（撰写） | 大纲 + 调研保留 | 只重做初稿 |
| 校对发现必修项涉及正文 | 模块 4（撰写） | 调研 + 大纲保留 | 重做受影响段落 → 重新排版 → 重新校对 |
| 校对发现必修项仅涉及排版 | 模块 6（排版） | 初稿保留 | 只重做排版 |
| 发布前确认不通过 | 当前环节 | 全部保留 | 按用户要求修改指定元素 |

**原则**：
- 回退时永远告诉用户"之前的工作已保存，只需重做 XX 部分"
- 永远只重做受影响的部分，不全盘重来
- 回退后自动把用户带到正确的环节，不需要用户自己想"我现在该干嘛"

### 7.4 跳环节前置校验

单模块调用时，编排器先检查前置依赖是否满足：

| 调用模块 | 前置要求 | 不满足时的提示 |
|---------|---------|---------------|
| 调研（模块2） | 需要选题 | "调研需要先有选题。告诉我你想写什么主题，或输入「自动选题」" |
| 大纲（模块3） | 需要调研报告 | "大纲需要调研素材。先运行调研，或直接粘贴你的素材给我" |
| 撰写（模块4） | 需要大纲 | "撰写需要大纲。先运行大纲构建，或直接告诉我文章结构" |
| 配图（模块5） | 需要初稿或大纲 | "配图需要知道文章内容。先有初稿或大纲才能规划配图" |
| 排版（模块6） | 需要初稿 | "排版需要文章正文。先完成撰写，或直接粘贴你的初稿" |
| 校对（模块7） | 需要成品 HTML | "校对需要排版后的内容。先运行排版" |
| 发布（模块8） | 需要成品 HTML | "发布准备需要排版后的内容。先运行排版" |
| 复盘（模块9） | 需要已发布文章 | "复盘需要文章已发布。告诉我发布后的数据" |

### 7.5 中间产物持久化

- 每个模块完成后，ArticleProject 对象自动保存到本地文件（`/workspace/projects/<article_id>/article.json`）
- 崩溃或中断后，用户重新启动时编排器自动检测未完成的项目并提示："检测到一篇未完成的文章《标题》，当前停在 XX 环节，要继续吗？"
- 用户可随时说"查看我的文章列表"看到所有进行中/已完成的项目及状态
- 任何环节的产出都可在 `/workspace/projects/<article_id>/` 下找到对应文件，不丢失
