# 公众号 AI 行业内容创作全流程

> **聚焦 AI 行业**：产品、技术、科普、行业解读，未来扩展 AI+ 方向。

## 核心架构

```
主编排器（本文件） → 10 个子模块（references/） + 配置层（assets/）
```

- **主编排器**（本文件）：流程控制、状态管理、关键节点门控、错误处理
- **子模块**：每个环节一个独立文档，在 `references/` 目录下
- **配置层**：默认信源、风格档案、排版模板，在 `assets/` 目录下，开箱即用

## MVP 分层

> **当前交付版本**：v1 核心 + v2 增强（模块1-8 已就绪），v3 闭环待定（模块9 复盘为可选）。

### v1 核心流程（首发）
模块1 选题 → 模块2 调研 → 模块3 大纲 → 模块4 撰写 → 模块6 排版
目标：跑通"从选题到可粘贴 HTML"的核心价值
> 注：v1 阶段跳过模块5配图，排版模块自动走纯文字降级路径。配图在 v2 阶段加入。

### v2 增强流程
+模块5 配图 + 模块7 校对 + 模块8 发布分发
目标：补齐视觉资产和质量把关

### v3 闭环流程
+模块9 复盘反馈 + 配置层个性化
目标：数据闭环和持续优化

## 全流程工作流

### 首次使用引导

编排器检测到 /workspace/projects/ 下无任何项目时，主动引导：
"👋 欢迎首次使用！我可以帮你完成公众号文章从选题到发布的全流程。
 你可以：
 → 给我一个选题关键词，我帮你跑全流程
 → 输入「自动选题」，我帮你看看 AI 行业最近有什么热点
 → 输入「只做 XX」，单独使用某个功能（如只排版、只调研）"

### 启动：创建文章项目

1. 用户给出选题关键词或请求自动选题
2. 创建文章项目，分配 ID，初始化 ArticleProject 对象（含 `schema_version: 1`）
3. 将项目状态保存到 `/workspace/projects/<article_id>/article.json`

### 流程串联

```
[模块1] 选题策划 → [模块2] 调研收集 → [模块3] 大纲构建 ★
  → [模块4] 正文撰写 ★ → [模块5] 配图制作 → [模块6] 排版美化
  → [模块7] 校对优化 → [模块8] 发布分发 ★ → [模块9] 复盘反馈（可选）
```

每个模块执行前：
1. 读取对应 `references/module-*.md` 获取详细指令
2. 从 `assets/` 加载所需配置（信源、风格、模板）
3. 执行模块逻辑
4. 将产出写入 ArticleProject 对象并持久化
5. 更新项目状态

### 关键节点门控

在 ★ 确认点处**必须暂停**，向用户展示产出并等待确认。

**每个确认点前必须先做产出非空校验**：若对应产出为空，不进入确认点，而是提示"XX 环节产出异常，正在重新生成"并自动重跑该环节。

**确认点 1（大纲确认）：**
- 前置校验：outline 不为空；为空时提示"大纲环节产出异常，正在重新生成"，不进入确认点
```
📋 大纲已完成，请确认：
  [展示大纲结构、各节要点、配图标注、差异化角度]
  可修改：结构/篇幅/配图标注/切入角度
  修改示例：修改：把第3节拆成两节 / 修改：配图改为数据图表
  → 回复「确认」继续撰写
  → 回复「修改：XX」调整大纲
  → 回复「重做」重新生成大纲
```

**确认点 2（初稿确认）：**
- 前置校验：draft 不为空；为空时提示"初稿环节产出异常，正在重新生成"，不进入确认点
```
📝 初稿已完成（约 XXXX 字），请确认：
  [展示初稿全文或摘要]
  可修改：调性/某段内容/术语处理/事实核查
  修改示例：修改：第2段语气更通俗 / 修改：GPT-4 首次出现加注释
  → 回复「确认」继续配图
  → 回复「修改：XX」调整初稿
  → 回复「重做」重新撰写
```

**确认点 3（发布前确认）：**
- 前置校验：formatted 不为空；为空时提示"排版环节产出异常，正在重新生成"，不进入确认点
```
📦 发布就绪包已准备好，请确认：
  成品HTML已保存到 `/workspace/projects/<article_id>/final.html`
  操作方式：在文件管理器中打开该文件，全选复制后粘贴到公众号编辑器
  [展示成品HTML预览链接、封面图、标题候选、摘要、标签、发布时机]
  可修改：标题/摘要/标签/封面/发布时机/HTML样式
  修改示例：修改：标题换成第二个候选 / 修改：标签加"AI Agent"
  → 回复「确认」完成流程
  → 回复「修改：XX」调整指定元素
```

### 回退规则

| 触发场景 | 回退到 | 保留 | 重做 |
|---------|-------|------|------|
| 大纲确认不通过 | 模块 3 | 调研报告 | 只重做大纲 |
| 初稿确认不通过 | 模块 4 | 大纲 + 调研 | 只重做初稿 |
| 校对发现必修项涉及正文 | 模块 4 | 调研 + 大纲 | 重做受影响段落→重新排版→重新校对 |
| 校对发现必修项仅涉及排版 | 模块 6 | 初稿 | 只重做排版 |
| 发布前确认不通过 | 当前环节 | 全部 | 按用户要求修改指定元素 |

**回退时必须告知用户**："之前的工作已保存，只需重做 XX 部分。"

### 调用模式自动判断

编排器根据用户输入自动判断调用模式：
- 用户给了完整选题/关键词 → 建议全流程
- 用户给了已有初稿/素材 → 建议从对应环节开始
- 用户明确说"只做XX" → 单模块调用

## 单模块独立调用

用户可跳过全流程，直接调用单个模块。编排器先执行前置校验：

| 调用模块 | 前置要求 | 不满足时的提示 |
|---------|---------|---------------|
| 调研（模块2） | 需要选题 | "调研需要先有选题。告诉我你想写什么主题，或输入「自动选题」" |
| 大纲（模块3） | 需要调研报告 | "大纲需要调研素材。先运行调研，或直接粘贴你的素材给我" |
| 撰写（模块4） | 需要大纲 | "撰写需要大纲。先运行大纲构建，或直接告诉我文章结构" |
| 配图（模块5） | 需要初稿或大纲 | "配图需要知道文章内容。先有初稿或大纲才能规划配图" |
| 排版（模块6） | 需要初稿 | "排版需要文章正文。先完成撰写，或直接粘贴你的初稿" |
| 校对（模块7） | 需要成品 HTML | "校对需要排版后的内容。先运行排版" |
| 发布（模块8） | 需要成品 HTML | "发布准备需要排版后的内容。先运行排版" |
| 复盘（模块9） | 需要已发布文章 | "复盘需要文章已发布。告诉我发布后的数据" |

## 错误处理规范

### 统一错误呈现（三段式，大白话）

每个错误信息必须包含：

```
❌ 【发生了什么】XX 环节：XX 出了问题
   【为什么】可能的原因（用大白话）
   【你该做什么】下一步操作建议
```

**原则：**
- 永远用大白话，不用技术术语
- 永远给一个可操作的下一步
- 能自动恢复的，恢复后告知用户"已自动处理"
- 不能自动恢复的，明确告诉用户怎么办

### 各模块降级方案

| 模块 | 失败点 | 自动降级 | 用户感知 |
|------|-------|---------|---------|
| 选题策划 | Web 搜索失败 | 用固定信源和 aihot 扫描热点 | "网络搜索暂不可用，已用内置信源选了 3 个热点" |
| 调研收集 | Web 搜索失败 | 用固定信源和 aihot 完成调研 | "Web 搜索暂不可用，已用其他通道完成调研" |
| 调研收集 | aihot 不可用 | 跳过实时资讯通道 | "实时资讯源暂不可用，已用其他通道完成调研" |
| 调研收集 | 某信源抓取失败 | 跳过该信源 | "XX 信源暂不可访问，已跳过，其余正常" |
| 正文撰写 | 事实核查无引用 | 标记"待核实"继续 | "第 3 段数据未找到出处，已标注待核实" |
| 配图制作 | guizang 依赖缺失（Node/Playwright） | 封面降级为 AI 生成，数据卡降级为 matplotlib | "社交卡片通道依赖 Node + Playwright，当前环境不满足，已降级处理" |
| 配图制作 | guizang 渲染失败 | 封面降级为 AI 生成，数据卡降级为 matplotlib | "社交卡片渲染失败，已降级为其他通道" |
| 配图制作 | AI 生成失败 | 文字占位 + 截图指引 | "封面图生成失败，已留占位符，可手动上传" |
| 配图制作 | 用户未提供截图 | 跳过该图，标注位置 | "产品截图需要你提供，已标注位置，不阻塞流程" |
| 配图制作 | 数据图表失败 | 降级为文字表格 | "图表生成失败，已用文字表格替代" |
| 排版美化 | 配图缺失 | 留空或文字占位 | "第 2 张配图缺失，已留占位，发布前可补图" |
| 发布分发 | 无历史数据 | 用通用时段建议 | "暂无历史数据，建议工作日早 8 点或晚 8 点发布" |
| 复盘反馈 | 用户未提供数据 | 跳过复盘 | "暂无数据，跳过复盘。7 天后告诉我阅读数据" |

## 项目持久化与恢复

### 存储结构

```
/workspace/projects/
  └── <article_id>/
      ├── article.json          # ArticleProject 完整状态
      ├── research.md           # 调研报告
      ├── outline.md            # 大纲
      ├── draft.md              # 初稿
      ├── final.html            # 成品 HTML
      └── images/               # 配图文件
```

### 崩溃恢复

用户重新启动时，编排器自动检测 `/workspace/projects/` 下未完成的项目：

```
📂 检测到 1 篇未完成的文章：
  《GPT-5 技术解读》— 当前停在「大纲确认」环节
  → 回复「继续」从大纲确认继续
  → 回复「查看列表」查看所有文章项目
  → 回复「新建」开始新文章
```

### 用户可用的指令

| 指令 | 作用 |
|------|------|
| 「查看我的文章列表」 | 列出所有进行中/已完成的项目 |
| 「继续」 | 从上次中断处继续 |
| 「重新做 XX 环节」 | 回退到指定环节重做 |
| 「跳过 XX」 | 跳过当前环节（如跳过配图） |
| 「只做 XX」 | 单模块调用 |
| 「帮助」 | 在任何时刻查看当前可用的所有操作和指令 |

> 编排器支持自然语言指令识别。以下说法均等效：
> - 「查看我的文章列表」「我的文章」「文章列表」「有哪些文章」→ 列出项目
> - 「继续」「接着做」→ 从断点继续
> - 「跳过配图」「不要图片」→ 跳过指定环节
> - 「只做排版」「只排版」→ 单模块调用
> 不需要精确匹配指令原文，用自然语言描述即可。

### 存储管理

- 已完成超过30天的项目，编排器提示用户归档
- history.json 限制保留最近50篇文章的复盘数据
- 单篇文章图片数量建议≤15张，超出时提示用户
- 关键节点（确认点通过后）自动生成 article.json.bak 备份
- 操作日志写入 projects/<id>/operations.log

## 配置层

### 加载顺序

每个模块启动时按以下顺序加载配置：

1. 检查 `/workspace/projects/<article_id>/config/` 是否有项目级覆盖配置
2. 检查 `/workspace/config/wechat-ai-writer/` 是否有用户级自定义配置
3. 加载 `assets/` 下的内置默认配置

**配置缺失时绝不中断流程**，自动使用默认值并标注"使用默认配置"。

### 配置文件

| 文件 | 位置 | 用途 |
|------|------|------|
| 默认信源清单 | `assets/default-sources.json` | AI 行业 25+ 权威信源（含 aihot 实时资讯） |
| 默认风格档案 | `assets/default-style-profile.md` | AI 行业调性 v2.0（活人感写作指南） |
| 默认排版模板 | `assets/default-template.html` | 公众号 HTML 模板 |
| 用户风格档案 | `/workspace/config/wechat-ai-writer/style-profile.md` | 用户个性化风格（可选） |
| 用户信源清单 | `/workspace/config/wechat-ai-writer/sources.json` | 用户自定义信源（可选） |
| 历史经验 | `/workspace/config/wechat-ai-writer/history.json` | 复盘模块自动写入 |
| 社交卡片 skill | `/workspace/skills/guizang-social-card-skill/` | 模块5社交卡片通道依赖（guizang，HTML+CSS+Playwright 渲染）。仓库内 `deps/guizang-social-card-skill/` 包含完整源码 |
| schema 版本 | ArticleProject.schema_version | 数据结构兼容性标识 |

### 如何个性化风格档案

1. 复制默认风格档案：`assets/default-style-profile.md`
2. 保存到 `/workspace/config/wechat-ai-writer/style-profile.md`
3. 按自己的偏好修改用词、句式、人称等
4. 下次运行时自动加载你的风格档案

同理可个性化信源清单（`sources.json`）和排版模板（`template.html`）。

### 配置合并策略

用户级自定义配置与默认配置的合并规则：

| 配置项 | 合并策略 | 说明 |
|--------|---------|------|
| 信源清单 | **合并** | 用户信源与默认信源合并，去重。用户可通过在信源中设置 `"override": true` 覆盖同名默认信源 |
| 风格档案 | **替换** | 用户风格档案存在时，完全替换默认风格档案（不合并） |
| 排版模板 | **替换** | 用户模板存在时，完全替换默认模板 |
| 历史经验 | **仅追加** | 复盘模块自动写入，不手动编辑，不覆盖 |

> 信源清单是唯一采用合并策略的配置——这样用户新增信源不会丢失内置的 25 个权威信源。

## 关键原则

1. **开箱即用**：所有配置有默认值，首次使用直接跑通，不强制预配置
2. **关键节点可控**：大纲、初稿、发布前三个点必须暂停等用户确认
3. **错误自解释**：所有错误三段式呈现，大白话，给可操作的下一步
4. **失败不阻塞**：每个失败点都有降级方案，能继续就继续，不能继续就明确告知
5. **回退不全盘重来**：只重做受影响的部分，之前的工作保留
6. **单模块可独立调用**：每个环节都能单独使用，前置依赖不满足时给明确提示
7. **进度可恢复**：每步自动存盘，崩溃后能从断点继续
8. **AI 行业特化**：竞品差异化、风格辨识度、数据图表、事实核查前置、发布后闭环

## 版本兼容

- ArticleProject 数据结构含 `schema_version` 字段（当前为 1）
- skill 升级时若数据结构变更，编排器检测旧版本项目时自动迁移或提示"该项目由旧版本创建，建议新建项目"
- 向后兼容：新版本能读取旧版本项目，旧版本无法读取新版本项目
