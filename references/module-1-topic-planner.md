# 模块 1：选题策划（Topic Planner）

> 热点追踪 + 六镜头角度发现 + HKR 质检。从 AI 行业热点中找到值得写的选题，并找到独一无二的切入角度。

## 模块职责

从 AI 行业热点中识别值得写的选题，通过六镜头角度发现法找到差异化的切入角度，为每个选题生成完整的内容简报，输出可供后续模块（调研、大纲、撰写）直接使用的选题方案。

聚焦领域：产品、技术、科普、行业解读四类 AI 内容。

## 输入

| 输入项 | 说明 | 是否必填 |
|--------|------|---------|
| 领域关键词 | 用户指定的 AI 细分方向，如「多模态」「Agent」「具身智能」 | 可选 |
| 空输入 | 用户未指定关键词时，自动扫描全局 AI 热点 | 自动 |

## 输出

选题方案文档，写入 `ArticleProject.topic` 字段。

每个选题方案包含：

| 子字段 | 说明 |
|--------|------|
| `title` | 暂定标题（1-2 个候选） |
| `content_type` | 文章类型：`tech_analysis` / `product_review` / `science_explainer` / `industry_insight` |
| `core_argument` | 核心论点，一句话说清本文想传达什么 |
| `target_audience` | 目标读者画像 |
| `estimated_length` | 预计篇幅范围 |
| `research_directions` | 重点调研方向 |
| `winning_angle` | 获胜角度描述（经六镜头发现 + 四项测试筛选后确定） |
| `angle_lens` | 获胜角度来源镜头（reframe / tension / hidden_cost / leading_indicator / category_error / counter_case） |
| `hooks` | 3 个开头钩子候选 |
| `pro_arguments` | 正方论点列表（含来源） |
| `con_arguments` | 反方论点列表（含来源） |
| `key_stats` | 5-8 个关键数据点（含来源） |
| `anchor_story` | 锚定故事时刻（一个具体的、能引发共鸣的场景或案例） |

## 执行步骤

### 1. 三路扫描热点

- **固定信源通道**：加载 `assets/default-sources.json` 中的固定信源，按关键词（用户给定或自动推断）匹配相关信源进行扫描。
- **Web 搜索通道**：使用 `WebSearch` 工具发起 Web 搜索，补充固定信源未覆盖的近期 AI 热点。
- **实时资讯通道**：通过 aihot（aihot.virxact.com）拉取近期 AI 热点日报和精选条目，获取结构化的每日 AI 动态。使用方式：WebFetch `https://aihot.virxact.com/api/daily` 获取今日日报，或 `https://aihot.virxact.com/api/items?category=products&days=7` 按分类和时间窗口拉取条目。
- 识别近期（建议覆盖近 7–14 天）出现的 AI 热点，形成候选选题池。

### 2. HKR 质检与打分筛选

对每个候选选题先过 **HKR 质检法**，再按四维度打分综合排序，筛出 Top 3–5。

**HKR 质检法**（三个维度交集才能成选题）：

| 维度 | 检查问题 | 不通过的信号 |
|------|---------|-------------|
| H 热度 | 这个话题现在有多少人在讨论？有没有搜索量？ | 没人关心，只有源信息没有讨论 |
| K 知识增量 | 这篇文章能给读者带来什么新东西？新角度、新数据、还是新整合？ | 纯转述，读者看了等于没看 |
| R 可行性 | 能不能写？素材够不够？时间来得及吗？有没有独家信息源？ | 素材不足，或已有大量同类文章填满 |

> **选题交集模型**：只在 H、K、R 三者交集区域选题。只有热度没有增量 = 追热点水文章；只有增量没有热度 = 自嗨；只有可行性没有热度和增量 = 浪费时间。

通过 HKR 质检的选题，再按以下四个维度打分，综合排序后筛出 Top 3–5：

| 维度 | 说明 |
|------|------|
| 时效性 | 热点发生时间、是否处于发酵期或衰减期 |
| 话题热度 | 讨论量、传播范围、跨平台热度 |
| 受众价值 | 对目标读者的信息增量与实用价值 |
| 差异化空间 | 现有内容是否已充分覆盖、本文能否做出增量 |

### 3. 六镜头角度发现

对每个入选选题，运行六镜头角度发现法，找到差异化的切入角度。

**前置步骤：对话图谱扫描**

在生成角度之前，先搜索现有讨论（WebSearch + aihot），了解人们实际怎么讨论这个话题。找到"引力中心"——也就是大多数人都在说的角度。六镜头的目标就是从引力中心之外开始。

**六个镜头**（每个镜头生成一个候选角度）：

| 镜头 | 核心问题 | 示例 |
|------|---------|------|
| **重新定义（Reframe）** | 把话题放到一个意想不到的框架里重新审视 | "AI 编程助手"不重新定义为"工具"，而是"同事" |
| **张力冲突（Tension）** | 找到话题中的内在矛盾或对立 | 效率提升了，但开发者反而更焦虑了 |
| **隐藏代价（Hidden Cost）** | 揭示大家没注意到的负面影响或成本 | 免费 AI 工具的隐性成本：你的代码变成了训练数据 |
| **先行指标（Leading Indicator）** | 把这个话题当作更大趋势的信号 | Claude Code 的 credit 池模式预示着 AI 工具从订阅制走向按量计费 |
| **分类错误（Category Error）** | 指出大家把这事归错了类 | Trae 不是 IDE，是 AI 原生开发平台 |
| **反例验证（Counter-Case）** | 找反例来挑战主流观点 | 大家都说 AI 编程助手提效，但有些场景它反而拖慢你 |

**角度生成规则**：
- 每个镜头生成一个候选角度，用一句话描述
- 候选角度必须与"引力中心"显著不同
- 候选角度必须有具体的事实或案例可以支撑（不能是纯空想）

### 4. 四项测试筛选

对六个候选角度逐个跑四项测试，选出唯一获胜角度。

| 测试 | 检查问题 | 不通过的信号 |
|------|---------|-------------|
| **独特性测试** | 这个角度是否与现有内容显著不同？搜一下，如果已有 3+ 篇同类角度文章，不通过 | 角度已被充分覆盖 |
| **支撑性测试** | 是否有足够的数据/案例/来源支撑这个角度？至少需要 2 个独立来源 | 素材不足，角度立不住 |
| **共鸣性测试** | 目标读者是否会关心这个角度？能否触发"对，我也这么觉得"或"等等，真的吗？" | 读者无感 |
| **可行性测试** | 在当前素材和时间约束下能否写好？是否需要超出范围的调研？ | 素材够不到，写不完整 |

**筛选规则**：
- 优先选择通过全部四项测试的角度
- 如果多个角度通过全部四项，选独特性得分最高的
- 如果没有角度通过全部四项，选通过 3 项的，并标记未通过项供后续补强
- 将获胜角度及其来源镜头记入 `winning_angle` 和 `angle_lens` 字段

### 5. 生成内容简报

基于获胜角度，生成完整的内容简报，供后续模块直接消费：

**5.1 三个开头钩子（hooks）**

生成 3 个不同风格的开头钩子候选，参照风格档案 v2.0 的开头必杀技：
- 叙事启动型：从一个具体的当下场景切入
- 荒诞事实型：直接抛出一个让人"？"的事实
- 好奇心驱动型：用一个反直觉的问题开场

**5.2 正反方论点（pro_arguments / con_arguments）**

围绕获胜角度，列出：
- 正方论点 2-3 个（支撑角度的论据，每个含来源）
- 反方论点 1-2 个（反对角度的论据，每个含来源）

正反方论点确保文章有对立面理解，不是单向输出。

**5.3 关键数据（key_stats）**

列出 5-8 个与角度相关的关键数据点，每个标注来源。数据需可在调研阶段验证。

**5.4 锚定故事（anchor_story）**

构思一个具体的、能引发共鸣的场景或案例，作为文章的情绪锚点。要求：
- 必须是具体的（有时间、地点、人物或产品名）
- 必须与获胜角度直接相关
- 必须能引发读者"我也遇到过"或"这会发生在谁身上"的共鸣

### 6. 输出选题方案

将完整选题方案写入 `ArticleProject.topic` 字段，包含所有子字段。向用户展示时，重点呈现：

- 暂定标题
- 获胜角度 + 来源镜头
- 三个钩子候选
- 正反方论点摘要
- 锚定故事

## 降级方案

- **Web 搜索失败**：只用固定信源和 aihot 扫描热点，并向用户告知「网络搜索暂不可用，已用内置信源选了 N 个热点」。六镜头的对话图谱扫描降级为仅基于固定信源内容判断。
- **aihot 不可用**：跳过实时资讯通道，使用固定信源和 Web 搜索完成扫描，告知用户「实时资讯源暂不可用」。
- **固定信源加载失败**：仅依赖 Web 搜索和 aihot，并告知用户信源加载异常。
- **候选选题不足 3 个**：如实告知热点偏少，放宽时间窗口或扩大关键词范围重新扫描。
- **六镜头均未通过四项测试**：选择通过 3 项的角度，标记薄弱项，告知用户「该角度在 XX 测试上偏弱，建议调研时重点补强」。

## 注意事项

1. **不凭空编造热点**：所有候选选题必须有信源或搜索结果支撑，无法溯源的选题剔除。
2. **时效优先但不追过期热点**：已明显衰减的热点降权，除非有新进展或新角度。
3. **差异化是核心**：打分时「差异化空间」权重应高于纯热度，避免扎堆已写烂的题材。
4. **角度优先于话题**：六镜头角度发现的价值在于"同一个话题，不同的切入角度"。如果所有角度都跟现有内容雷同，应该换话题而不是硬写。
5. **对话图谱扫描不可跳过**：六镜头的前提是知道"引力中心"在哪——也就是大多数人正在说什么。跳过这一步，六镜头就变成了闭门造车。
6. **标题仅为暂定**：最终标题在发布模块（模块 8）确定，此处不强求完美。
7. **尊重用户意向**：用户给定关键词时，扫描结果应围绕该关键词，不要跑题到无关热点。
8. **输出可被直接消费**：选题方案的字段需与后续模块期望一致，避免结构缺失导致流程中断。内容简报中的 hooks、pro/con arguments、key_stats、anchor_story 应可被模块 3（大纲）和模块 4（撰写）直接引用。
