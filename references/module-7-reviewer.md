# 模块 7：校对优化（Reviewer）

> 成品 HTML 质量把关：文字 + 可读性 + 合规。事实核查已在撰写环节（模块 4）前置完成，本模块不再重复事实核查。

## 模块职责

- 对成品 HTML 做文字校对（错别字、标点、语法、术语一致性）
- 做可读性优化，给出具体修改建议（而非泛泛而谈）
- 做合规检查，规避 AI 行业内容常见风险点
- 对问题分级（必须修 / 建议修 / 可选优化）
- 无严重问题直接通过；有问题按异常流回退对应模块

## 输入

| 输入项 | 来源 | 说明 |
|--------|------|------|
| 成品 HTML | `ArticleProject.formatted` | 排版后的成品，校对对象 |
| 风格档案 | `assets/default-style-profile.md`（或用户级） | 用于术语一致性、调性校验参考 |
| 大纲 | `ArticleProject.outline` | 校验正文是否覆盖大纲要点（可读性辅助参考） |

> 事实核查已在模块 4 撰写环节前置完成，本模块输入不含独立的事实核查数据源，遇疑似事实问题转交模块 4 处理。

## 输出

校对报告写入 `ArticleProject.review` 字段，结构如下：

```json
{
  "status": "passed | needs_fix",
  "issues": [
    {
      "level": "must_fix | suggested | optional",
      "category": "文字 | 可读性 | 合规",
      "location": "问题位置（段落/小节定位）",
      "description": "问题描述",
      "suggestion": "具体修改建议"
    }
  ],
  "fixes_applied": [
    "已应用的修改记录（可直接修的轻量问题）"
  ]
}
```

- `status: passed` 表示无必须修问题，可直接进入发布
- `status: needs_fix` 表示存在必须修问题，触发异常流回退

## 执行步骤

### 步骤 1：文字校对

逐段检查：

- **错别字**：同音字、形近字误用
- **标点**：中英文标点混用、全半角混用、引号书名号规范
- **语法**：病句、主谓搭配、长句结构
- **术语一致性**：同一概念全篇用词是否统一（如「大语言模型 / 大模型 / LLM」是否混用），参照风格档案

可直接修的轻量问题（错别字、标点）记入 `fixes_applied` 并直接修正；涉及语义的改动只给建议不擅改。

### 步骤 2：可读性优化

给出具体修改建议，避免「建议优化表达」这类空话：

- **信息密度检查**：是否堆砌术语、是否一段塞太多要点
- **长段拆分建议**：超长段落给出拆分点建议
- **过渡生硬检测**：段落间逻辑断层，给出衔接建议
- **结构完整性**：正文是否覆盖大纲要点，有无遗漏或跑题

每条建议需写明「在哪、什么问题、怎么改」。

### 步骤 3：合规检查

AI 行业内容重点规避以下风险，措辞需谨慎：

- **避免模型能力夸大**：不用「最强」「完全超越人类」「无所不能」等绝对化表述
- **避免未经验证预测**：对未来趋势的判断需有依据或标注不确定性，不写断言式预言
- **AI 安全 / 监管话题措辞谨慎**：涉及安全风险、监管政策时客观中立，不渲染恐慌、不轻下结论
- **避免误导性对比**：竞品对比基于调研数据，不歪曲、不拉踩
- **敏感信息脱敏**：检查正文是否残留内部系统名、API key、未公开数据

### 步骤 4：问题分级

将所有问题按严重程度分级：

| 级别 | 含义 | 处理方式 |
|------|------|---------|
| `must_fix`（必须修） | 错别字影响理解、合规风险、严重语病、事实性嫌疑 | 触发异常流回退 |
| `suggested`（建议修） | 可读性优化、术语统一、过渡衔接 | 列入报告，建议修但不阻塞 |
| `optional`（可选优化） | 表达润色、排版微调 | 列入报告，由用户决定 |

分级后汇总：无 `must_fix` → `status: passed`；有 `must_fix` → `status: needs_fix`。

## 降级方案

- **无严重问题直接通过**：`issues` 中无 `must_fix` 级别，`status` 置为 `passed`，进入发布模块
- **轻量问题自动修**：错别字、标点等可直接修的，记入 `fixes_applied` 并修正，不阻塞
- **无法判定的问题**：标注「待核实」转入建议级，不擅自改为必须修

## 异常流

校对发现 `must_fix` 问题时，按问题归属回退对应模块，不全盘重来：

| 问题归属 | 回退到 | 保留 | 重做 |
|---------|-------|------|------|
| 问题涉及正文（语病、合规、事实嫌疑、可读性） | 模块 4 | 调研 + 大纲 | 重做受影响段落 → 重新排版（模块 6） → 重新校对（模块 7） |
| 问题仅涉及排版（格式错乱、图片错位、引用样式） | 模块 6 | 初稿 | 只重做排版 → 重新校对 |

回退时必须告知用户：「之前的工作已保存，只需重做 XX 部分。」

### 修正同步规则

- **轻量修正（不触发回退）**：fixes_applied 中的修正直接应用到 `ArticleProject.formatted.html`，同时同步到 `ArticleProject.draft.content_markdown`，保持两者一致
- **触发回退时**：回退到模块4重写正文后，draft 重新生成，模块6重新排版生成新的 formatted，模块7之前的 fixes_applied 自动失效，需重新校对。此时 review 字段保留历史记录供参考，但 issues 状态标记为 `superseded`（已被覆盖）
- **原则**：任何时候 draft 和 formatted 必须保持内容一致，修正不能只改一边

## 注意事项

1. **不重复事实核查**：事实核查是模块 4 的前置职责，本模块遇疑似事实问题转交模块 4，不在校对环节临时查证
2. **建议要具体**：每条可读性建议必须写明位置、问题、改法，禁止「建议优化」「表达欠佳」这类无效反馈
3. **合规是底线**：能力夸大、未验证预测、安全监管措辞是 AI 行业内容的高风险点，宁可保守不可激进
4. **分级要准确**：必须修的判定从严（影响理解 / 合规风险 / 事实嫌疑），表达润色归可选优化，避免过度拦截阻塞流程
5. **不擅改正文语义**：可直接修的仅限错别字、标点等无歧义轻量问题；涉及语义、观点的改动只给建议
6. **回退精准**：按问题归属回退对应模块，正文问题回模块 4、纯排版问题回模块 6，不全盘重来
7. **校对报告可追溯**：`issues` 与 `fixes_applied` 都要记录清楚，便于用户查阅和复盘模块分析
