# 模块 9：复盘反馈（Retrospective）

> 发布后收集数据，分析表现，沉淀经验回喂选题和发布模块，形成闭环。

## 模块职责

- 收集文章发布后的表现数据（用户手动提供）
- 按内容类型、选题来源、标题风格、发布时段分析最佳组合
- 沉淀经验，回喂选题模块、发布模块、风格档案
- 长期追踪数据趋势，识别读者偏好变化
- 写入历史经验到配置层，供后续创作复用

复盘是全流程的闭环环节：把发布后的真实表现数据，转化为下一轮创作的优化依据。

> **复盘为可选环节，不阻塞流程**：文章发布后流程即视为完成。复盘模块仅在用户主动提供数据时才运行；未提供数据时跳过，并提示「发布 7 天后可随时告诉我阅读数据，我帮你分析表现」。复盘从「必经流程」调整为「可选增强」。

## 输入

| 输入项 | 来源 | 说明 |
|--------|------|------|
| 文章项目 ID | `ArticleProject.id` | 定位待复盘的文章项目 |
| 文章主题与类型 | `ArticleProject.topic` | 内容类型（content_type），用于归因分析 |
| 发布信息 | `ArticleProject.publish` | 实际使用的标题风格、标签、发布时间，用于归因分析 |
| 发布后数据 | 用户手动提供 | 阅读量、完读率、转发数、留言数（发布 7 天后提供） |
| 历史经验 | `/workspace/config/wechat-ai-writer/history.json` | 已有经验库，用于趋势对比与增量更新 |

发布后数据由用户手动提供，本模块不自动抓取公众号后台数据。

## 输出

- 复盘报告写入 `ArticleProject.retrospective` 字段，结构如下：

```json
{
  "status": "completed | skipped",
  "data": {
    "views": 0,
    "read_completion_rate": 0.0,
    "shares": 0,
    "comments": 0
  },
  "analysis": {
    "best_combination": "内容类型+选题来源+标题风格+发布时段的最佳组合",
    "weak_signals": ["表现低于预期的信号"]
  },
  "feedback": {
    "to_topic_planner": "选题评分权重调整建议",
    "to_publisher": "发布时机与标题建议优化",
    "to_style_profile": "调性偏好更新建议"
  },
  "trend_note": "读者偏好变化观察"
}
```

- 同时更新配置层历史经验 `/workspace/config/wechat-ai-writer/history.json`
- `status: skipped` 表示用户未提供数据，跳过本次复盘

## 执行步骤

### 步骤 1：数据收集

- 提示用户在文章发布 7 天后提供数据：阅读量、完读率、转发数、留言数
- 用户提供后记录到 `retrospective.data`
- 用户未提供 → 走降级方案，跳过复盘

### 步骤 2：表现分析

结合本次数据与文章元信息（内容类型、选题来源、标题风格、发布时段），分析最佳组合：

- 按内容类型对比表现（产品 / 技术 / 科普 / 行业解读哪类更受读者欢迎）
- 按选题来源对比（热点追踪 vs 深度解读 vs 系列连载）
- 按标题风格对比（直述型 / 悬念型 / 数字型哪类点击更好）
- 按发布时段对比（早 8 / 晚 8 / 周末等哪类时段完读率、转发更高）
- 识别表现低于预期的弱信号（如完读率低可能是正文过长或开头不够吸引）

### 步骤 3：经验沉淀

将分析结论回喂三个模块：

- **回喂选题模块（模块 1）**：根据表现数据调整选题评分权重（如某类选题完读率高则提权）
- **回喂发布模块（模块 8）**：优化发布时机建议与标题风格推荐（如悬念型标题点击更高则优先推荐）
- **回喂风格档案**：更新调性偏好（如读者对某种表达方式反馈更好则纳入风格档案）

回喂以建议形式写入 `retrospective.feedback`，并实际更新 `history.json` 中的经验权重。

### 步骤 4：趋势追踪

长期追踪数据趋势，识别读者偏好变化：

- 对比本次数据与历史均值，判断是高于/低于/持平
- 识别长期趋势（如某细分领域阅读量持续上升、某类标题逐渐失效）
- 趋势观察写入 `retrospective.trend_note`，供后续创作参考

### 步骤 5：写入历史经验

- 将本次复盘数据与结论增量写入 `/workspace/config/wechat-ai-writer/history.json`
- 历史经验文件结构包含：按内容类型/选题来源/标题风格/发布时段聚合的表现统计与权重建议
- 写入后告知用户「经验已沉淀，将用于后续创作优化」

## 降级方案

| 失败点 | 自动降级 | 用户感知（三段式） |
|--------|---------|------------------|
| 用户未提供数据 | 跳过复盘 | 「暂无数据，跳过复盘。7 天后告诉我阅读数据」 |
| 数据不完整（缺部分指标） | 用已有数据分析，缺项标注 | 「XX 指标缺失，已用现有数据分析，结论仅供参考」 |
| 历史经验文件不存在 | 自动创建并初始化 | 「未找到历史经验库，已自动创建，本次作为首条记录」 |
| 历史经验文件读取失败 | 仅做本次分析，不更新历史 | 「历史经验读取失败，本次仅做单篇分析，未更新经验库」 |

## 注意事项

1. **数据需用户手动提供**：本模块不自动抓取公众号后台，阅读量/完读率/转发数/留言数由用户提供，提示在发布 7 天后提供以获取稳定数据
2. **回喂是闭环关键**：复盘的价值在于把数据转化为下一轮创作的优化依据，feedback 三项（选题/发布/风格）不可省
3. **分析看组合不看单点**：单篇数据波动大，应按「内容类型+选题来源+标题风格+发布时段」组合归因，避免单篇偶发结论误导
4. **历史经验增量更新**：写入 `history.json` 是增量累积，不覆盖已有经验，长期形成趋势判断
5. **数据不完整不硬凑**：缺指标时用已有数据分析并标注局限，不编造缺失数据
6. **文件不存在自动初始化**：首次复盘时 `history.json` 不存在属正常情况，自动创建初始化，不报错阻塞
7. **复盘可延迟**：用户暂无数据时跳过，不强制即时复盘，提示下次提供即可，尊重发布后的数据稳定周期
8. **结论标不确定性**：基于有限样本的分析结论应标注置信度，小样本不写断言式规律
9. **复盘数据为用户手动提供，不设强制时间窗口**：用户可在发布后任意时间提供数据触发复盘，不强制 7 天时限，尊重用户节奏
